# これは

- Reactコンポーネントの切り方や、それに付随するCSSクラスの設計について、極力悩まないための俺ルール集でありポエムである
- まだルールが未熟で、カバーできてない部分があったり、理想論すぎるケースもあると思う
- これは個人プロダクト用のルールだが、明文化しておくことで会社での共同作業時に「私は普段こうやってますが、チームではどうします？」のタネにしたい


# コンポーネントの切り方とCSS設計

- 基本的にatomic designに則ってディレクトリ構造を作り、その下に該当する粒度のコンポーネントのディレクトリを作り、その下に同名のjs,css,画像素材などを配置する

  - 具体的には、 `atoms` `molecules` `organisms` `templates` ディレクトリを作り、それぞれの下に対応する粒度のコンポーネントを配置する
    - `pages` まではなくていいと思う。多いと複雑になるし、コンポーネントの観点だと最上位レベルはtemplateだと思っている
    - 近いコンポーネントが多い場合は一段ディレクトリを掘ったりする。 `molecules/category` など
  - css modulesは使わず、コンポーネントと同名のクラス名を割り当てる。コンポーネント中の要素にはBEMのElementであるとして命名を行う
    - `HogeComponent` なら、rootは `HogeComponent` 、その要素は `HogeComponent__Title` , `HogeComponent__Body` のように
    - cssのimportは、必ずjsファイル中で他のコンポーネントのimportのあとで行う。
      - 利用される側より利用する側のほうがより具体的なコンポーネントであるはず。このルールでimportすると、結果的に上方向から下方向に向かってだんだん具体的になっていくcssが出来上がる
      - 指定が衝突しても(font指定など)、使う側=具体的な側に倒れる。
  - 同じクラス名をコード中で使いまわさない、二度参照しない
    - 使いまわすのはコンポーネント、使いまわしたくなるということはそのクラス指定に意味がある、その意味の単位でコンポーネント化するべし
    - 例えば多くのボタンに `.featureButton` クラスを降る場合、 `FeatureButton` コンポーネントに切り出し、 内容物を `children` で指定すると言うかたちでコンポーネント化できるはず
    - 画像素材についても同じく、例えば `CategoryIcon` `SectionFrame` などのatomとして切り出せるはず。複数箇所から参照する必要はない
  - それぞれのコンポーネントは、共通でclassName porpertyをもたせ、利用コンポーネントから修正可能にする。
    - このclassNameは、コンポーネントのroot要素のclassNameにそのまま渡す
    - `SpecialPanelComponent__SuperLinkButton` のように、親コンポーネント視点で子コンポーネントはElementであるとして、BEMに則ってそういう命名をする
    - これは位置調整にしか用いない。あるコンポーネントを自身のどの位置に配置したいかは、親が知るべきであるため
      - 位置調整のプロパティ･･･ position, top系、margin系、flex-chilren用のプロパティなど
    - 逆に言うと、それぞれのコンポーネントは自身のroot要素に位置に関するスタイル指定は行わない。
      - 自分がどういう位置に使われるかまで知っているのはおかしい
      - また、親からのclassName指定で不必要に衝突し、動作が予測しづらくなるのを避けたい
    - 親から位置情報以外をカスタマイズしたい場合は、以下のmodifierを使う
  - 位置情報と関係ない外部からのカスタマイズは、個別のカスタマイザブルなプロパティを用意するか、modifierプロパティを利用する
    - 例えば、更新履歴のいち要素を表す `HistoryItem` コンポーネント、 `new={true}` で新規要素の強調表示をできるようにする
      - `ComponentName--new`
    - もしくは、 `modifier="new"` という指定もアリである。上記のように具体的にプロパティ名が決めづらい場合に使う
      - BEMのmodifierと同じであり、実際このmodifierは `ComponentName--modifier` 、 `ComponentName__ElementName--modifier`  のように機械的にクラス命名を行う
      - 上記をやりやすくするような withModifierみたいなutil関数を作っとくと便利
      - 全部の子要素に対して上記のようなことはせず、カスタマイズで変更されうる要素のみでOK
    - このルールの優れているところは、外部から見た目に関するstyle指定を注入できるようにしたとして、その指定は指定先コンポーネントのどの子要素に適応されるのか悩まなくて良くなる点
      - 適切なmodifierさえ指定すれば、あとはコンポーネントの責任でカスタマイズされるべき子要素にmodifierが振られ、スタイルが適応される
    - 具体的なプロパティ名が思いつかず、変にtypeのような謎Propertyを生やすくらいなら、このルールに則りmodifierを使う
    - modifierは配列での複数指定も許す。 `['new', 'active']` のような
  - 積極的に内部stateless componentによる切り出しを行う
    - 意味のある単位での分割を心がける。それらがatom やmoleculesにもなるはず。
    - ただし、とは言え他のコンポーネントから使わんだろというパーツは、同ファイル内でexportせずに、内部コンポーネントとして作ってよい。わざわざatoms下などへ切り出して可読性を落とす必要はない
      - また、子コンポーネントが `li` で親コンポーネントが `ul` など、不可分なケースもあるはず。そういう場合もこのルールは役立つ
    - 内部コンポーネントの名前は、多くの場合 `ParentComponentItem` `ParentComponentButton` のように、親の名前が接頭になるはずだが、これに限らずより良い命名がある場合はそれを用いる


# 副作用やロジックとの分離

  - 上記の規則はロジックや副作用について無視している
  - ロジックや副作用はできるだけ見た目がなく、再利用可能なコンポーネントという形で分離しておき、それを他のComponentから利用する

    - 例えば、クリックするとモーダルを開くボタンは `OpenModalButton` として切り出し、単一のbutton要素から構成され、 `className` プロパティや `children` プロパティによって外部から自由にViewを与えることができるように作る

      - この場合、classNameによる外部からの位置情報以外のプロパティの修正を許す

        - あくまでこの場合は該当ComponentはViewに感心がないということでの例外ケース
    - また、childrenを全く取らず、見た目もない機能だけのコンポーネントという切り方もありである

      - Scrollをハンドリングし、reduxにactionをdispatchする、 `ScrollHandler` など切ったりする
    - どちらのケースも、class によるComponentを使う。逆に、本ルールでは副作用を伴わない殆どをstateless componentで記述する
  - reduxのconnectを用いたロジックとViewの切り離しも有効である

    - mapStateToProps, mapDispatchToPropsにロジックを集中させ、コンポーネント本体はstateless componentで記述する
    - ロジックがでかい場合、これらの関数を切り出してテスタブルにするなど考えられる
  - これで対応できないケースも多々あると思うが、それらをまとめて扱えるパターンを示すには経験が浅い

    - 直近だとrefをどう分離するかで悩まされた
